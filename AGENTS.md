# AGENTS.md — 開発・拡張時の方針

このドキュメントは、本リポジトリでコードや素材を追加する**人間／AI エージェント**が必ず読むためのものです。
PR を出す前にこれと矛盾していないか確認してください。

---

## 1. ゲーム性の原則

- このゲームは **「攻撃で倒す」より「回避・記録・通報・連携」で勝つ**。
- 敵 HP をゼロにする勝利条件は実装しない。勝利は常に以下のいずれか：
  - 通報ゲージ満タン → ROUND 進行
  - 店外誘導ゲージ満タン or 敵が画面右の EXIT を超える
- プレイヤーの能動アクションのうち、**直接攻撃にあたるもの（パンチ・キック・武器）は実装しない**。Z はあくまでガード／距離取り。
- 「報連相が一番強い」ことが手触りで伝わる調整を最優先。攻撃で殴るほうが速い、と感じさせない。

---

## 2. トーンと表現の禁止事項

以下は **実装・セリフ・素材いずれにも入れない**：

- 差別語・ヘイト表現（人種・性別・国籍・職業・身体的特徴等）
- 実在企業／実在店舗／実在個人への名誉毀損や直接的な揶揄
- **未成年の性的表現**。家出中の若者キャラを足す場合も「成人または年齢不詳の困窮した若者」として扱う
- 弱者を一方的に笑う構図（ホームレス風キャラを「ただ叩かれる対象」として扱わない、等）
- 過度な暴力描写（出血、骨折、武器による殺傷、医療的に深刻な描写）
- 実在ロゴ・実在キャラの完全コピー。仮称（LOWSON / NIGHT MART / 歌舞伎町24h 等）に留める

代わりに、笑いの軸は次に絞る：

- 深夜ワンオペ接客の理不尽さ
- 接客マニュアルと現実のズレ
- 「殴るより通報のほうが強い」というメタ

セリフ追加時は `src/game/dialogue.ts` に集約する。新規キャラを追加する場合も同ファイルへ追加し、表現方針をレビューする。

---

## 3. MVP を壊さない

現在の MVP は次の 4 状態を行き来できる：

```
title  →  round1  →  round2  →  clear
                 ↘            ↘
                  gameover    gameover
```

新機能を追加する場合、

- 上記の最低限のフローが従来通り動くことを必ずローカルで確認
- 既存ゲージ（STRESS / REPORT / EVICT / ANNOY）の意味を変更しない（追加は OK）
- 既存キーバインドを既存意味から変更しない（追加は OK）
- 「敵を殴って HP を削る」設計に逸脱させない

---

## 4. まず動くものを優先、その後リファクタ

- 機能追加は「ナイーブに動かす → 動いたら整理」の順で。最初から完璧な抽象は要らない。
- 1 機能 = 1 ファイルでよい。`src/game/` 直下に追加して OK。
- 不安定な実装でもまず `npm run dev` で目視確認できる状態を作る。
- 抽象化は同じパターンが 3 か所以上に出てから。早すぎる DRY は避ける。
- ゲーム定数は新規追加分も `src/game/config.ts` に集約。マジックナンバーをコードに散らさない。

---

## 5. ビジュアル方針

- **当面は仮素材のままでよい**。矩形・簡易ピクセルでゲーム性を最初に固める。
- 生成 AI で出力した画像をゲーム内素材として直接使わない。出すならデザイナーがピクセル化する前提。
- 解像度は内部 384×216 を維持。フォントは `MS Gothic` / `monospace` の系統で、ピクセル感を保つ。
- 背景でネオン看板を増やす場合、**実在店名は使わない**。

---

## 6. テスト・確認手順

PR 前に以下を最低限通す：

```bash
npm install
npm run build         # 型エラーが無いこと
npm run dev           # 手動で:
                      #  - タイトル → ROUND1 → ROUND2 → CLEAR
                      #  - つかまれて X 連打で抜ける
                      #  - 距離を取って通報ゲージが伸びる
                      #  - 近距離だと通報ゲージが減る
                      #  - GAME OVER → Enter でリトライ
```

UI 変更時は実際にブラウザで操作して、**ゴールデンパス（タイトル→クリア）と、つかみ脱出のような失敗パスの両方**を踏んでから完了報告する。

---

## 7. PR の書き方

- スコープを 1 機能に絞る。バランス調整とビジュアル変更は分ける。
- 表現に関する変更（セリフ追加、敵キャラ追加）は本ドキュメント §2 と矛盾しないか説明欄に明記する。
- 配信や SNS で映えそうなネタを盛りたくなった時ほど、§2 の禁止事項に戻ること。
